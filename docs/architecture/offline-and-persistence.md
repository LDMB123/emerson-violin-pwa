# Offline, Persistence, And Service Worker

The app treats local storage as the primary runtime, not a fallback nicety. Persistence and offline behavior should be updated together.

## Persistence Model

- JSON and blob persistence flow through `src/persistence/storage.js`
- IndexedDB is the preferred store
- localStorage remains the fallback path for environments or operations where IndexedDB is unavailable or fails
- collection-backed keys and blob storage are handled below the public storage facade so feature code can keep a stable API

## Offline Model

- Service worker capability checks start in `src/platform/sw-support.js`
- install/update/offline flows live under `src/platform/`
- the shipped service worker lives in `public/sw.js`
- precache asset lists are generated by `scripts/build-sw-assets.js`

## Source Of Truth

- Storage facade and fallback behavior: `src/persistence/storage.js`
- IndexedDB implementation details: `src/persistence/storage-idb.js`
- service worker runtime logic: `public/sw.js`
- offline/update/install controllers: `src/platform/`

## Maintainer Guidance

- If a change affects offline launch, persistence survivability, or service worker registration, update both the relevant guide and this architecture note
- Keep docs focused on guarantees and ownership, not browser-specific snapshot counts
- Scratch restore paths like `_archived/original-assets/...` are local workflow details, not tracked repo guarantees
